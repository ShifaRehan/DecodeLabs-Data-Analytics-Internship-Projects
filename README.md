## 🔍 Project Deep-Dive

<details>
<summary><b>🧹 Project 01 — Data Cleaning & Preparation</b> (click to expand)</summary>
<br>

**The problem:** Real-world data is never clean. It has missing values, duplicate rows, inconsistent text formatting, and outright errors.

**What I did:**
- Identified and handled missing values (rather than blindly deleting them)
- Removed duplicate records that would have skewed every later analysis
- Standardized inconsistent formats (dates, text casing, categories)
- Prepared a final, analysis-ready dataset used as the foundation for every project after this one

**Why it matters:** This is the least glamorous step in data analytics and the most important. A stunning chart built on dirty data is still a wrong chart. This project is the quality-control gate everything else passes through.

**Tools:** Python, Pandas, NumPy, Jupyter Notebook

</details>

<details>
<summary><b>📊 Project 02 — Exploratory Data Analysis (EDA)</b> (click to expand)</summary>
<br>

**The problem:** A clean dataset is still just a table of numbers until someone actually looks at what it's telling them.

**What I did:**
- Used descriptive statistics (mean, median, distribution, correlation) to summarize the dataset numerically
- Built visualizations with Matplotlib to spot trends, seasonal patterns, and anomalies
- Investigated outliers to understand whether they were genuine business signals or data errors

**Why it matters:** EDA is where a data analyst forms their *first hypotheses*. Before you can tell anyone a story about the data, you have to understand what's actually happening inside it.

**Tools:** Python, Pandas, Matplotlib, Jupyter Notebook

</details>

<details>
<summary><b>🗄️ Project 03 — SQL Data Analysis</b> (click to expand)</summary>
<br>

**The problem:** Real companies don't store their data in a single Excel sheet they store it in databases. Being able to *query* data directly is a non-negotiable analyst skill.

**What I did:**
- Loaded the cleaned dataset into **MySQL**
- Wrote SQL queries to evaluate product performance and customer behavior
- Extracted specific, targeted business insights instead of exploring everything at once
- Converted query results into actionable recommendations

**Why it matters:** SQL is how analysts talk directly to a company's data infrastructure. This project simulates being handed a business question and having to go get the answer yourself, straight from the source.

**Tools:** MySQL, SQL

</details>

<details>
<summary><b>📈 Project 04 — Sales Analytics Dashboard (Power BI)</b> (click to expand)</summary>
<br>

**The problem:** Insights that live inside a notebook or a query result never reach the people who make decisions. Someone has to turn analysis into something a non-technical executive can understand in five seconds.

**What I did:**
- Designed and built a full interactive **Power BI dashboard** on cleaned e-commerce order data
- Applied real design principles: the **Chart Selection Matrix**, **Data-Ink Ratio**, elimination of "chartjunk," and the **5-second rule**
- Built KPI cards, a donut chart, bar charts, a trend line, a funnel chart, and interactive slicers
- Surfaced a genuinely important business finding: a **41.4% order cancellation/return rate**, putting **$520K of the $1.26M in total revenue at risk**

**Why it matters:** This is the "so what" step. Everything from Projects 1–3 — cleaning, exploring, querying exists to build toward a moment like this: a single dashboard a business owner can actually use to make a decision.

**Tools:** Power BI Desktop, Power Query, DAX

📄 **[Read the full Project 04 README →](./Project%20-%2004/README.md)** for the complete dashboard walkthrough, design philosophy, and data insights.

</details>

## 🌱 Reflections

This internship was structured so that no project stood alone each one forced me to build on the output of the last.
